1. Executive Summary
RedArky RedArky is an async, multi-agent platform that monitors Reddit, Hacker News, GitHub Trending, and RSS feeds on topics you define. It runs a production-grade data pipeline to collect, clean, classify, and embed raw social data into a queryable data lake and vector store. An AI agent system then analyses the stored data to find recurring pain points, identifies market gaps (places where problems exist but good solutions don't), generates mini technical specifications for potential SaaS products, and optionally engages with Reddit communities to validate ideas — all before delivering a structured morning digest.

This is simultaneously a resume-grade engineering project and a genuinely useful personal tool. It demonstrates five distinct engineering disciplines in one cohesive system rather than five disconnected toy projects.

3. Services Architecture
RedArky is composed of five independently deployable services. Each lives in its own directory in the monorepo with its own Dockerfile, requirements/go.mod, and tests.

RedArky-scraper  |  Go 1.22+
Description
Standalone Go microservice. Uses goroutines for concurrent HTTP scraping across all sources. Exposes a single internal REST endpoint POST /scrape that accepts a mission config and returns a batch of structured JSON records. Implements a token-bucket rate limiter per source domain, circuit breaker pattern for flaky external APIs, and exponential backoff on retries.
Tech stack
Go 1.22 · net/http · goroutines · sync.WaitGroup · golang.org/x/time/rate · PRAW adapter · Algolia API · RSS parser

RedArky-api  |  Python 3.12 / FastAPI
Description
Core application server. Handles user authentication (JWT + refresh tokens), full CRUD for Missions and Agents, triggers Airflow DAG runs via the Airflow REST API, serves a WebSocket endpoint for real-time agent log streaming to the frontend, and exposes REST endpoints for all dashboard data. Also runs Celery Beat for mission schedule management.
Tech stack
FastAPI · Pydantic v2 · SQLAlchemy 2.0 · Alembic · Celery + Beat · JWT auth · WebSocket · httpx · python-praw

RedArky-pipeline  |  Python 3.12 / Apache Airflow 2.9
Description
Orchestrates the full ETL/ELT DAG per mission. Raw JSON from the scraper lands in S3 first (immutable source of truth). Airflow then runs 8 sequential tasks: validate → clean → deduplicate → classify → upload Parquet → generate embeddings → load PostgreSQL/pgvector → trigger agents. Runs on a per-mission schedule (hourly, daily, etc.). Every task is idempotent and can be safely retried.
Tech stack
Apache Airflow 2.9 · boto3 / S3 · pyarrow / Parquet · Pydantic · OpenAI embeddings · pgvector · Redis dedup set · Celery

RedArky-agents  |  Python 3.12 / LangGraph
Description
Multi-agent pipeline running as Celery workers. A LangGraph state graph defines the agent execution flow: Supervisor decides the plan, Researcher retrieves relevant data via pgvector similarity search, Gap Finder cross-references with web search to identify unmet needs, Spec Writer generates structured mini-specs, Engagement Agent optionally drafts Reddit interactions, and Report Agent produces the morning digest. Custom user-defined agents slot in after Researcher.
Tech stack
LangGraph · LangChain · Celery workers · OpenAI GPT-4o · pgvector RAG · PRAW (engagement) · Pydantic structured output · Redis pub/sub

RedArky-web  |  Next.js 14 / TypeScript
Description
Dashboard frontend. Mission builder with subreddit selector and agent configurator. Real-time agent log stream via WebSocket. Daily digest reader with gap cards and mini-spec viewer. Engagement approval queue (user must approve every Reddit action before it posts). Custom agent builder UI. Minimal and functional — this is not the star of the portfolio, the backend is.
Tech stack
Next.js 14 · TypeScript · React Query · Tailwind CSS · WebSocket client · Recharts · Zustand

5. Data Pipeline (Apache Airflow)
The pipeline follows an ELT pattern: raw data lands in S3 first as an immutable source of truth, transformations happen downstream. This means any transformation bug can be fixed and the pipeline replayed without re-scraping.

5.1  S3 data lake layout
s3://RedArky-lake/
  raw/                                 ← immutable raw API responses
    reddit/{mission_id}/{YYYY-MM-DD}/{batch_id}.json
    hn/{mission_id}/{YYYY-MM-DD}/{batch_id}.json
  processed/                           ← cleaned, classified Parquet
    {mission_id}/{YYYY-MM-DD}/{batch_id}.parquet
  dead-letter/                         ← failed validation records
    {mission_id}/{YYYY-MM-DD}/invalid_{batch_id}.json
  reports/                             ← daily digest output
    {mission_id}/{YYYY-MM-DD}/digest.json

5.2  Airflow DAG — mission_intelligence_dag
One DAG instance per mission, parameterised by mission_id. Triggered on schedule or manually. All tasks are idempotent.

5.3  Data quality strategy
Deduplication
SHA-256 hash of (url + title) stored in a Redis SET per mission. O(1) lookup before any write. Survives restarts and replays.
Schema validation
Pydantic model gate on every record. Failures go to dead-letter S3 prefix with structured failure reason — never silently lost.
Quality filter
Drops posts below configurable score threshold, deleted/removed content, posts under 50 characters, obvious spam patterns.
Idempotency
Every DAG task can be retried safely. S3 writes use deterministic batch_id keys. Postgres inserts use ON CONFLICT DO NOTHING.
Observability
Airflow task logs to CloudWatch. Failed tasks alert via email/Slack. Dead-letter queue monitored by a daily Airflow task.


6. AI Agent System (LangGraph)
All agents are Celery workers executing a LangGraph state graph. The Supervisor node reads the mission config and constructs the execution plan. Each subsequent agent reads from and writes to a shared Pydantic state object — typed, validated, and persisted to the agent_runs table after every node completes.

Supervisor  —  Orchestrator
What it does
Reads the mission config, the current batch signal (mission_id, batch_id, item_count), and any previously completed agent_runs. Constructs the LangGraph execution plan for this mission type — which agents run, in what order, with which conditional edges. Handles retry edges for each node.
Input / Output
IN:  Mission config + Redis batch signal + previous run history
OUT: LangGraph execution plan — ordered node list + conditional edges

Researcher  —  Data analysis
What it does
Embeds the mission goal string and runs a pgvector cosine similarity search against the latest batch of embeddings. Retrieves the top-K most relevant chunks. Groups findings by topic cluster using a lightweight clustering prompt. Extracts recurring themes, pain points, and sentiment patterns with frequency counts.
Input / Output
IN:  Mission goal embedding + pgvector index + latest batch data_items
OUT: themes[], pain_points[] with frequency + severity, sentiment_map

Gap Finder  —  Market analysis
What it does
Takes the pain_points list from the Researcher. For each pain point, uses the LangChain web search tool to find existing solutions. Compares: does an adequate solution exist? Ranks gaps by a composite score of (frequency × severity × absence_of_solution). Returns a list of validated market gaps with supporting evidence IDs.
Input / Output
IN:  Research output (pain_points, frequency_map) + web search tool
OUT: market_gaps[] ranked by composite score, each with evidence_ids[]

Spec Writer  —  Synthesis
What it does
For each market gap above the confidence threshold, generates a structured mini technical specification. Output is a Pydantic-validated JSON object stored in the market_gaps table. Includes: problem statement, proposed solution, rough architecture sketch (2-3 sentences), target user persona, monetisation hypothesis, and 3 validation questions for the Engagement Agent.
Input / Output
IN:  Market gap + supporting evidence text chunks from pgvector
OUT: spec_json per gap: {problem, solution, architecture, target_user, monetisation, validation_questions[]}

Engagement  —  Optional — user-activated
What it does
Uses PRAW with the user's Reddit OAuth credentials to draft a comment or validation question targeting a relevant thread identified by the Gap Finder. Writes the draft to the engagement_actions table with status pending_approval. NEVER posts without explicit user approval from the dashboard. Rate limited to 5 actions per day per mission.
Input / Output
IN:  Spec validation_questions + target thread URLs identified by Gap Finder
OUT: engagement_actions rows with status=pending_approval (human gate required before posting)

Report  —  Daily digest — 7am
What it does
Runs every morning at 07:00. Reads all agent_runs outputs from the past 24 hours for each active mission. Compiles a structured daily digest: new gaps found, mini-specs written, engagement results (impressions, replies), trend summary comparing this week vs last week. Writes Markdown + structured JSON to the reports table.
Input / Output
IN:  All Postgres agent_run outputs from past 24h for the mission
OUT: reports row: {content JSONB, markdown_body TEXT, gaps_found, specs_written, generated_at}

Custom (user-defined)  —  Extensible slot
What it does
User defines an agent via 4 fields in the dashboard: (1) data query — a pgvector similarity search query or raw SQL filter defining what data this agent reads; (2) prompt template with variables {data}, {mission_goal}, {date_range}; (3) output schema — a JSON Schema that the LLM is forced to match; (4) schedule — on pipeline completion, daily, or on demand. Runs after Researcher, before Report.
Input / Output
IN:  User-defined pgvector/SQL query against mission data
OUT: User-defined Pydantic output — stored in agent_runs.output_json

13. Interview Preparation
These are the exact questions you will be asked about this project in technical interviews. Read the answers out loud multiple times before any interview.

Q: Why Go for the scraper instead of Python asyncio?
A: Go's goroutines handle 200+ concurrent HTTP requests with 10–20× less memory than Python threads. I benchmarked both — Go scraped 1,000 Reddit posts in ~3 seconds vs ~12 seconds with Python asyncio on the same machine. More importantly, the Go service is stateless and compiles to a single binary, making it trivially easy to containerise and scale horizontally. The scraper is also a natural service boundary — it has one job (fetch raw data) and Go's performance characteristics are ideal for that job.

Q: Explain your data pipeline architecture.
A: I use an ELT pattern — raw data lands in S3 first as an immutable source of truth. This is a deliberate architectural decision: if I write a bug in the classification logic in week 4, I can fix it and replay the entire pipeline from S3 without re-scraping Reddit. Airflow then runs 8 sequential tasks: validate with Pydantic, deduplicate via a Redis hash set (O(1)), classify with GPT-4o-mini, convert to Parquet and upload to S3, batch embed, then bulk upsert into pgvector. Failed records go to a dead-letter prefix — never silently dropped. Every task is idempotent so safe to retry.

Q: How do the AI agents communicate and share state?
A: I use a LangGraph state graph. Each agent is a node that reads from and writes to a shared Pydantic state object — fully typed and validated at each step. The Supervisor node reads the mission config and constructs the execution plan. Each downstream agent receives typed output from the previous one. If an agent fails, the graph has retry edges. The final state is persisted to the agent_runs table after each node, so I can inspect any run in the dashboard and see exactly what each agent received and produced.

Q: How would you scale this to 10,000 active missions?
A: All services are stateless and scale horizontally. The Go scraper and Celery workers add nodes trivially. Airflow already distributes tasks across a worker pool. I'd partition missions across Celery queues by priority tier (paid vs free). The real bottleneck is LLM API throughput — I'd add a token-bucket rate limiter per tier and cache embeddings aggressively since many missions overlap in content. For the database, I'd add a read replica for agent queries and keep writes to the primary. pgvector IVFFlat performance degrades above ~1M vectors — I'd move to HNSW index or partition embeddings by mission at that scale.

Q: Why ELT and not ETL? Why S3 before Postgres?
A: Two reasons. First, S3 is the replay layer — raw data is immutable and I can reprocess from any point. If I change a transform or find a bug, I don't have to re-scrape. Second, Parquet on S3 lets me run analytical queries cheaply (DuckDB locally, Athena in production) without hammering Postgres. Postgres holds the structured, indexed, hot query data. S3 holds the historical archive. Different tools for different access patterns — that's the core of data engineering.

Q: What happens when the OpenAI API is down?
A: The generate_embeddings and classify_content DAG tasks have exponential backoff with jitter — up to 5 retries over ~2 minutes. Airflow marks tasks as retrying, not failed, during this window. If the API is down for longer, Airflow marks the task failed and sends an alert. The previous successful run's data remains in the DB — the system degrades gracefully (no new data processed) rather than corrupting existing data. For production I'd add a fallback to a local embedding model (e.g. sentence-transformers) for the embedding task specifically.


14. Pitfalls to Avoid
Do not use Twitter/X API
Costs $100+/month with no meaningful benefit over Reddit + HN at personal scale
Do not build the frontend first
The backend is what impresses engineers. Defer the dashboard to Phase 6
Do not skip the dead-letter queue
Silently dropping failed records is an anti-pattern. It hides data quality issues
Do not over-engineer the agents
Start with simple LLM calls, validate the output quality, then add LangGraph orchestration
Do not post to Reddit without approval queue
Automated posting without human gate risks ban and is ethically problematic
Do not use a single monolithic Python service
Separation of RedArky-api, RedArky-pipeline, RedArky-agents is intentional — it demonstrates distributed systems thinking
Do not skip Alembic migrations
Raw SQL schema scripts are not production practice. Migrations are a required signal of engineering maturity
Do not hardcode credentials
Every secret goes in .env / K8s Secrets / AWS Secrets Manager from day one
